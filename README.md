# IR System

一个面向苏州大学导师信息的轻量检索系统，提供 CLI 与 GUI 两种使用方式。

## 更新与功能

- 多次迭代查询: 先尝试精确短语匹配, 若无结果再放宽为分块/词项检索.
- 放宽查询条件: 例如 "机器翻译" 可自动降级为 "机器" 或 "翻译" 的组合检索.
- 可选模糊查询: 当存在 `fuzzywuzzy` 依赖时, 进行字符串模糊匹配以提升召回; 主界面的字段后过滤也会同步使用模糊匹配, 避免把模糊命中的结果误删.
- 字段限定检索: `姓名: / 研究方向: / 论文:` 前缀可将检索范围限定到对应字段, 提升精确度.
- GUI 可调阈值: 模糊检索阈值可在界面中从 `0-100` 调整, 默认 `70`, 数值越低越宽松.
- 结果清洗去冗余: 自动剥离语料元数据头(doc_id/url 等)、裁剪页面模板与页脚(基本信息/版权所有 等)、折叠多余空白, 并对研究方向/简介/论文做长度截断.
- 片段智能去重: 当片段内容与已展示字段重复时自动隐藏.
- 命中关键词高亮: 仅保留最长非重叠匹配词(如 "周国栋" 不再附带 "周国"/"栋" 碎片).
- 更稳健脱敏: 邮箱需带合法域名(避免误伤 "@Google Scholar"), 电话仅匹配手机号/带分隔符座机(避免误伤课题编号与年份区间).
- **中英文跨语言检索**: 输入 `NLP`/`ML`/`events extraction`/`GNN`/`5G` 等英文缩写或短语, 自动扩展为中文等价词检索语料, 效果与搜「自然语言处理」「事件抽取」等基本一致; GUI 快捷查询已加入英文示例.

## 功能概览

- TF-IDF 词项打分与结果排序.
- 支持姓名/研究方向/论文等信息检索.
- GUI 过滤器: 姓名/研究方向/论文条件组合; 当启用模糊检索时, 过滤阶段也会使用模糊匹配.
- GUI 增强: 快捷查询、基础/优化模式切换、Top-K 可调、按相关度/姓名/学院排序、状态栏耗时反馈、一键打开教师主页。
- GUI 检索模式: 支持开关放宽检索、开关模糊检索, 并可调整模糊阈值。
- GUI 美化: 卡片式结果、序号徽章、相关度徽章、字段标签对齐、命中关键词彩色标签、分隔线与滚轮滚动。
- 基础 vs 优化并排对比: 点击「基础 vs 优化 对比」按钮, 弹出双栏窗口同屏比较两套配置的结果, 标注各自命中数/耗时, 并用绿色「优化新增」标签高亮优化模式额外召回的导师。

## 多次迭代检索（实现细节）

下面是项目中“多次迭代”检索在代码层面的实现要点与执行顺序，便于调试与定制：

- 调用入口：`search(query, docs, teachers, inverted, doc_norms, ...)`（`ir_system.py`）。
- 执行顺序（通用检索路径）：
  1. 精确短语匹配：`_phrase_search(query, ...)` — 在文档与教师拼接文本中查找归一化后的完整短语，按出现次数计分并优先返回。
  2. 放宽检索（分块/词项）：先用 `_relax_terms(query)` 生成宽松词项/子串，再用 `_token_search(query_tokens, ...)` 在倒排索引上累加权重召回（基于 `(1+log(tf))*idf` 的局部权重，除以 `doc_norms` 归一化）。
  3. 常规 token 检索：对原始 `query_tokens` 再做一次 `_token_search`（作为普通 TF-IDF 路径）。
  4. 可选模糊兜底：若 `enable_fuzzy=True`，调用 `_fuzzy_search(query, ..., threshold)`（使用字符串相似度，如 `partial_ratio`），阈值由 `fuzzy_threshold` 控制。
- 其他路径：字段限定（`论文:`/`研究方向:`）由 `_field_search` 专门处理；姓名直达通过 `teacher_lookup` 快速返回。
- 去重与排序：所有候选通过 `_dedupe_and_rank()` 按教师/学院合并、稳定排序并截取 Top-K。

示例（query="机器翻译"）：优先尝试短语 "机器翻译"，若未命中则降级为分词/二字片段检索（放宽），仍无结果且启用模糊时再以相似度匹配近似文本。

调优建议：

- 若需更严格命中可把 `_token_search(..., require_all=True)` 用于特定场景。
- 中文检索质量可通过替换 `_tokenize` 为 `jieba` 分词来显著提升（减少碎片化）。
- 排序可考虑引入 BM25 替代或补充当前 TF-IDF 得分以获得更稳健相关度。

## 放宽检索 vs 模糊检索（解释与建议）

本项目同时实现了两类不同的“召回放宽”策略，理解二者差别有助于在 GUI/评测中选择合适的配置：

- 放宽检索（Relax / Term Relaxation）:

  - 定义：把原始查询按规则拆解或降级为更宽松的词项/子串组合再检索，例如把“机器翻译”拆为“机器”与“翻译”的组合或二元切分再匹配。
  - 特点：基于规则，结果可预测、噪声较小，适合解决“查询过严导致无命中”的问题。
  - 实现说明：本系统先尝试精确短语匹配，失败后会降级为分块/词项检索以提高召回。
- 模糊检索（Fuzzy / Similarity-based）:

  - 定义：用字符串相似度（如 partial ratio）来判断查询与文档是否“近似匹配”，不要求完全子串相等。
  - 特点：容错性强（拼写、次序、部分不一致仍能命中），但更容易召回不相关结果（噪声）。
  - 实现说明：实现依赖第三方库（`fuzzywuzzy` 或可替换为 `rapidfuzz`），通过阈值控制（0–100）。

推荐策略与 GUI 行为：

- 默认执行顺序（系统内置）：先尝试精确短语 → 放宽检索（分块/词项）→ 可选模糊检索兜底。
- GUI 已提供开关：可分别启用/禁用“放宽检索”和“模糊检索”，并在“检索模式”中通过滑块/数值框调整模糊阈值（默认 70，数值越低越宽松）。
- 注意：主界面在做字段过滤（例如按研究方向/论文过滤）时会同步使用模糊匹配（当模糊检索开启），避免把只有模糊命中的结果在后处理阶段误删。

### 模糊阈值（`fuzzy_threshold`）详解

- 定义：模糊阈值是字符串相似度函数（如 `partial_ratio`/`ratio`）返回的分数下限，取值范围 0–100。仅当相似度分数 >= 阈值时，候选才被视为匹配并返回或保留在后处理过滤中。
- 取值含义（建议分段）：
  - `100`：等同于严格子串/完全匹配，仅接近完全一致的字符串命中（极高精确度，极低召回）。
  - `90–99`：非常接近匹配，能容忍极小拼写/空白差异（高精度、低召回）。
  - `75–89`：中高相似度，覆盖常见轻微变形或词序差异（常用折中区间）。
  - `50–74`：宽松匹配，返回较多近似或部分重合的结果（召回↑、噪声↑）。
  - `0–49`：非常宽松，会显著增加不相关命中，通常只在特殊离线分析或宽泛搜集时使用。
- 对检索效果的影响：阈值越高 → 精确度↑、召回↓；阈值越低 → 召回↑、精确度↓。选择阈值即是在精确度与召回率之间权衡。
- 性能与实践：模糊匹配开销较大，建议先用倒排索引或分阶段检索把候选集限制到较小集合，再在这些候选上做模糊比对（当前实现即采用这种分阶段策略）。
- 调优建议：
  - 默认起点：`70–75`（折中）；
  - 若需容错小拼写或标点，尝试 `85–95`；
  - 需更宽泛召回时可下调到 `60–70`，但注意噪声增加；
  - 在中文场景，改用 `jieba` 分词通常能改善相似度判定，从而允许更低阈值仍保持可接受噪声。
  - 用 `evaluate.py` 做阈值 sweep（例如 50/60/70/80/90）观测 Precision/Recall 曲线并据此选择。
- GUI 建议：在 `fuzzy_threshold` 控件附近显示简短提示（例如“模糊阈值 0–100，越高越严格；默认 70”），并在高级设置中提供“快速评估”按钮做阈值对比。

权衡建议：

- 首选放宽检索以提升召回并保持较低噪声；仅在确实需要容错（拼写/同义/非精确词）时启用模糊检索。
- 若启用模糊检索，建议从 `70` 开始调优，观察 `evaluate.py` 的 `hit@k` 与响应时间，再根据噪声水平调低或调高阈值。

性能与质量提升方向：考虑对中文使用 `jieba` 做更鲁棒的分词，或引入 BM25 排序以替代/补充当前的 TF-IDF 打分，从而在不依赖模糊匹配的前提下提高中文检索表现。

### 阈值扫描（拼写/近似查询示例）

本仓库在 `evaluate.py` 中新增了一个便捷的阈值扫描功能，用于对少量带拼写或近似错误的查询在不同 `fuzzy_threshold` 值下观测检索效果和 Top-K 命中。主要用途是帮助选择合适的模糊阈值（见“模糊阈值详解”节）。

- 内置的拼写/近似测试样例（示例）：

  - `机器翻yi`（拼写/拼音混入）
  - `机z械翻译`（插入噪字符）
  - `机器fanyi`（拼音混入）
  - `机器翻譯`（繁体字变体）
- 使用方法：在项目根目录运行以下命令（会为每个阈值生成单独 CSV）：

```bash
# 使用默认阈值序列 50,60,70,80,90
python evaluate.py --threshold-sweep

# 指定阈值序列例如 60,70,80
python evaluate.py --threshold-sweep --thresholds 60,70,80

# 同时调整 Top-K
python evaluate.py --threshold-sweep --top-k 3 --thresholds 50,60,70
```

- 输出：每个阈值生成 `outputs/eval_threshold_{t}.csv`（包含每条示例查询的 top-k 结果与命中标记），并会把阈值扫描的汇总写入你指定的 `--summary-out` 文件（默认 `outputs/eval_summary.csv`）。
- 建议：先用 `--top-k 5` 做几次试跑，观察 `hit@1` 与 `hit@k` 随阈值变化的趋势，再决定在 GUI 中设置的默认值。

* **定义** : `top-k`：返回结果中排名靠前的前 `k` 个条目（例如 `top-5` 返回评分最高的 5 条）。
* **在评测中的作用** : 用于计算 `hit@1`（期望在第1位命中）与 `hit@k`（期望在前 k 位任一命中）。
* **指标含义** : `hit@1` 测量精确度最高的位置；`hit@k` 测量在更宽松的候选窗口内的召回能力。
* **对结果与性能的影响** : 增大 `k` → 召回↑（更容易命中期望项），但会增加 IO/展示与用户浏览成本；模糊匹配在大 `k` 下更容易把噪声排进返回集合。
* **选取建议** : GUI 展示常用 `k=5` 或 `10`；离线评测按业务场景固定（比如 `k=1/5/10` 做多点对比）；快速调试用较小 `k` 以节省时间。
* **如何设置** : 在 [evaluate.py](vscode-file://vscode-app/d:/Programs/Microsoft%20VS%20Code/6a44c352bd/resources/app/out/vs/code/electron-browser/workbench/workbench.html) 中用 `--top-k`，例如：
* `python evaluate.py --top-k 5`
* 在阈值扫描中也可传：`python evaluate.py --threshold-sweep --top-k 3`。
* **实践提示** : 若关注「用户第一个看到的结果」，优先优化 `hit@1`；若关注「能否在前几条里找到正确答案」，关注 `hit@k` 并据此调整检索/阈值/排

## 大模型结构化抽取（crawler/llm_extract.py，可选增强）

各教师主页栏目命名差异极大（研究领域/Research/会议论文/期刊论文…），关键词匹配易漏。
该脚本把爬虫存的 `full_text` 整体交给大模型，统一抽取 研究方向/简介/代表论文/关键词，
不依赖固定栏目名；属“规则保底 + LLM 增强”：

- 发送前再次脱敏；输出严格 JSON 带容错解析；限速/重试/退避；缓存(已抽取跳过, `--force` 重抽)。
- 没有 API key 或调用失败时自动回退保留规则版结果, 不清空已有数据。
- 抽取后回写 `teachers.json` 并重建语料/索引, IR 系统可直接使用。

依赖与环境变量（DeepSeek，OpenAI SDK）:

- `pip install openai`
- `DEEPSEEK_API_KEY`(必填，兼容旧名 `LLM_API_KEY`)、`LLM_BASE_URL`(默认 `https://api.deepseek.com`)、`LLM_MODEL`(默认 `deepseek-v4-pro`)。
- 默认开启 DeepSeek V4 Pro 思考模式(`reasoning_effort=high` + `thinking.enabled`)，可用 `--no-thinking` 关闭以提速省钱。

```powershell
$env:DEEPSEEK_API_KEY="sk-xxx"
python crawler/llm_extract.py --limit 5      # 先小批量试抽
python crawler/llm_extract.py                # 全量抽取并回写
python crawler/llm_extract.py --dry-run      # 只打印不回写
python crawler/llm_extract.py --no-thinking  # 关闭思考模式
```

## 评测输出

- `evaluate.py` 会同时评测 baseline（不放宽、不模糊）与 optimized（放宽+模糊）两套检索配置。

- 汇总结果默认输出到 `outputs/eval_summary.csv`，可直接用于画柱状图/表格。
- 每次执行会把“本次做了什么 + 指标摘要”追加写入 `outputs/eval_run_log.md` 作为实验过程日志。

## 论文专项筛查（crawler/llm_papers_screen.py）

全量 `llm_extract.py` 抽取后，部分教师论文栏仍混入期刊缩写、统计句、审稿职务等噪音（如赵雷主页只有「在 ICDE/TKDE 发表论文」而无具体标题）。本脚本用**极短提示词**单独调用 DeepSeek，只鉴别真实论文并输出结构化字段：

- 返回 `title / venue / year / ccf_rank`，写入 `papers_struct` 与 `papers_text`
- 无具体标题时返回空数组并清空脏数据
- 标记 `papers_screened: true`，支持 checkpoint 与 Ctrl+C 安全保存

```powershell
python crawler/llm_papers_screen.py --name 赵雷          # 单人验证
python crawler/llm_papers_screen.py --suspicious-only   # 仅 14 条疑似噪音
python crawler/llm_papers_screen.py --all               # 全量未筛查记录
python crawler/llm_papers_screen.py --dry-run           # 预览不写回
```

日志：`crawled_data/llm_papers_screen_log.txt`

## 中英文跨语言检索（ir_system.py）

语料以中文为主, 但论文标题/主页常含英文。检索层在 `search()` 前通过 `QueryPlan` 做查询扩展:

| 类型 | 示例输入 | 扩展为 |
|------|----------|--------|
| 缩写 | `NLP` `ML` `GNN` `RAG` `LLM` `5G` `SDN` `FPGA` | 自然语言处理、机器学习、图神经网络… |
| 短语 | `events extraction` `machine learning` `recommendation system` | 事件抽取、机器学习、推荐系统… |
| 字段限定 | `research: NLP` `paper: event extraction` | 先剥前缀, 再扩展 |

实现要点:

- `_ABBR_ALIASES`: **149** 个独立英文单词 → 中文等价词
- `_EN_PHRASE_RULES`: **142** 条正则短语 → 中文
- 公开 API: `expand_query(query)`、`query_matches_text(text, query)`（GUI 侧栏过滤复用）
- 检索顺序不变: 扩展短语精确匹配 → 扩展词项倒排检索 → 模糊兜底; 高亮关键词同样走扩展词

验证样例（2026-06-10）:

```
NLP                  → 周国栋、李培峰…
events extraction    → 李培峰、周国栋
GNN                  → 周经亚、赵朋朋…
bioinformatics       → 朱斐、权丽君…
speech enhancement   → 陈雪勤
```

扩展表位于 `ir_system.py` 顶部 `_ABBR_ALIASES` / `_EN_PHRASE_RULES`, 可按领域继续追加.

### 二、检索与展示阶段

| 问题           | 表现                                       | 处理                                                                                 |
| -------------- | ------------------------------------------ | ------------------------------------------------------------------------------------ |
| 查询过严无结果 | 「机器翻译」等复合词查不到                 | 多次迭代查询：精确短语 → 分块/词项 → 可选 `fuzzywuzzy` 模糊匹配                  |
| 英文查不到中文 | 输入 `NLP`/`ML`/`events extraction` 无效果 | `QueryPlan` 跨语言扩展：149 缩写 + 142 短语规则; GUI 快捷查询加英文按钮          |
| 结果冗余难读   | 语料头、页脚版权、重复片段堆在卡片里       | `build_display()` 剥离元数据头、裁剪模板、片段去重、字段长度截断                   |
| 高亮碎片       | 「周国栋」同时高亮「周国」「栋」           | 仅保留最长非重叠匹配                                                                 |

| GUI 论文难展示 | 论文一大段纯文本，无 CCF 等级              | 引入 `papers_struct`（title/venue/year/ccf_rank）；GUI 逐条列表 + A/B/C 彩色 badge |

### 三、结构化抽取（LLM）阶段

### 四、论文字段质量（重点）

这是实验中反复出现、迭代最多的部分。

**4.1 典型坏数据类型**

| 类型               | 样例教师           | 坏数据示例                                                | 处理手段                                             |
| ------------------ | ------------------ | --------------------------------------------------------- | ---------------------------------------------------- |
| 纯期刊/会议缩写    | 赵雷、王中卿       | `ICDE`、`TKDE`、`类国际会议及期刊上发表论文`        | `llm_papers_screen.py` 专项筛查 → 清空            |
| 统计/职务句        | 权丽君、杜扬       | 「发表论文 N 篇」「程序委员会成员」                       | `_is_paper_line_noise` + LLM 筛查                  |
| 占位栏             | 杨璐、曹敏         | `软件著作\n专利`                                        | `batch_fix_teachers.py` 识别占位后清空或从全文提取 |
| dict 字符串        | 丁泓铭             | `{'title': '...', 'venue': '...'}` 写入 `papers_text` | `repair_papers_record()` 解析还原                  |
| 基金/课程/招生混入 | 李培峰（部分轮次） | 基金项目名、招生要求进论文栏                              | `sanitize_papers_record()` 多模式过滤              |

**4.2 处理策略演进**

1. **规则批量修复**（`batch_fix_teachers.py`，无需 API）：占位识别、正则从 `full_text` 抽英文论文标题、生成 `papers_struct`。
2. **LLM 全量抽取**（`llm_extract.py`）：331/332 成功，统一研究方向/简介/论文；内置 `sanitize_papers_record` 后处理。
3. **论文专项筛查**（`llm_papers_screen.py`）：提示词极短、职责单一；先 `--suspicious-only` 清 14 人噪音，再 `--all` 对 324 人全量结构化论文。

**4.3 论文筛查结果（2026-06-09/10）**

- 疑似噪音 14 人：13 人确认为 0 条（主页无具体标题）；**向德辉** 抽出 10 条真实论文（含 venue、ccf_rank）。
- 全量筛查 324 人：成功 324，失败 0；约 **210/339** 人有有效 `papers_struct`。
- **赵雷**：筛查前 `papers_text` 为 ICDE/TKDE 碎片 → 筛查后清空（符合主页仅有统计描述的事实）。

### 八、经验小结

1. **爬虫与 LLM 分工**：爬虫保证 `full_text` 完整不丢；LLM 负责语义归一；规则脚本做便宜的高精度过滤。
2. **论文比研究方向更难**：栏目差异大、中英文混排、统计句与真实标题难用单一正则区分 → 专项短 prompt 比全量抽取更稳。
3. **空字段优于脏字段**：赵雷类案例宁可论文为空，也不展示 ICDE/TKDE 缩写，避免检索与 GUI 误导。
4. **可复现**：各环节有 checkpoint、dry-run、限量参数；评测日志自动追加，便于报告引用。
5. **英文查询靠扩展表**：语料是中文时, 不能指望倒排索引直接命中 `NLP`; 维护缩写/短语映射表是性价比最高的跨语言方案。
